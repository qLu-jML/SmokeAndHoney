Introduction: The Karpathy Method

The Karpathy Method is an incremental validation approach developed by Andrej Karpathy for building complex systems: start small, validate rigorously, expand systematically. Rather than building a monolithic 800-line simulation and debugging it as a black box, we break Smoke & Honey's simulation into 8 independent phases, each producing a standalone Python script that validates one subsystem against real-world science.

The Phased Approach

This is fundamentally a validation-first methodology:

  • Phase N incorporates all outputs from Phases 1 through N-1
  • Each phase validates one biological subsystem with scientific references
  • Every test is written to match observable, measurable phenomena from real bee colonies
  • When Phase N passes, we carry forward ALL mathematical guarantees to Phase N+1
  • When Phase N fails, the isolated scope makes diagnosis straightforward
Key Insight: When all individual phases pass but integration fails (as with Phase 8), it tells us the subsystems are mathematically correct but calibration constants between systems need adjustment. This is far more actionable than debugging a tangled codebase.

Why This Matters

Traditional debugging of a complex simulation often involves:

  • Running the full system, observing wrong output
  • Not knowing which of 50 variables caused the problem
  • Guessing at fixes and hoping they propagate correctly
  • Regression testing becoming a nightmare

The Karpathy approach inverts this: we test each subsystem in isolation first, so when integration issues appear, we know exactly where to look. Each phase is a self-contained Python program that can be run instantly and independently.

Key Findings from Integration

Insight 1: No Impossible States - The fact that we get reasonable (if imperfect) output means the math is sound. We are not chasing silent errors or infinite loops; we are calibrating real parameters.

Insight 2: Environment-Forage Mismatch - Phase 6 (real Iowa flower data) produces much lower peak yields than the mock forage model in Phase 3. This is honest: real Iowa has seasonal gaps, and our mock model was optimistic. The integration reveals this mismatch immediately.

Insight 3: Compounding Stress Multipliers - Queen stress, Varroa impact, foraging pressure, and congestion all act multiplicatively in Phase 8, reducing overall fitness faster than expected. In isolation (Phases 4-5), each looks correct. Together, they create a "perfect storm" that kills colonies faster than observed.

Insight 4: Population Timing - The colony builds population, but peaks slightly lower and later than observed. This suggests either:

  • Laying rate response to congestion is too aggressive
  • Emergence rate is slightly delayed
  • Nurse investment in brood care is consuming too much time, reducing nectar storage

Calibration Roadmap

Priority 1: Forage Normalization (Phase 6)

  • Phase 6 produces absolute nectar amounts; Phase 8 expects normalized 0-1 forage levels
  • Scaling mismatch is causing Phase 3 mock to be inflated
  • Fix: Validate forage_level range [0, 1] in full integration

Priority 2: Queen Stress Multipliers (Phase 4)

  • Queen laying rate drops too much under congestion stress
  • In Phase 4 alone, tests pass; in Phase 8, interaction is too harsh
  • Fix: Re-check congestion_modifier calculation; expected threshold may need raising

Priority 3: Varroa Impact Curve (Phase 5)

  • Mite load grows correctly, but impact on survival is gentler than observed
  • Possible: mite-induced mortality is too low, or treatment assumptions are unrealistic
  • Fix: Cross-check published Varroa-mortality correlations

Priority 4: Winter Survival (Phases 3 + 5)

  • Colonies survive winter at too-high rates (~85% vs 60-70% observed)
  • Possible: honey consumption underestimated, or Varroa impact underestimated in winter
  • Fix: Validate winter consumption model against real beekeeping records
This is a Feature, Not a Bug: In a monolithic simulation, these tuning issues would be buried in 800 lines of interdependent code. Here, Phase 8 tells us exactly which prior phases need recalibration. We can fix one subsystem, re-run Phase 8, and verify the fix in minutes.

Carries Forward

  • Complete system validation framework
  • Calibrated parameter baseline for all subsystems
  • Actionable tuning roadmap for game balance

Running the Research (PowerShell Workflow)

Each phase has its own folder with a run.ps1 launcher and a README.txt with full instructions. Open PowerShell, navigate to the phase folder, and run the script. Work through phases in order -- each one builds confidence in the next.

General Workflow

cd research\karpathy\phase1_brood_biology
.\run.ps1

Each run.ps1 will show colored output: GREEN = all tests pass (move on), RED = failures remain (keep tuning). Read the README.txt in each folder for detailed stop criteria.

Run All Phases Sequentially (PowerShell)

From the project root, to run all phases in order:

$phases = @("phase1_brood_biology", "phase2_population", "phase3_foraging_honey",
"phase4_queen_comb", "phase5_disease_pests", "phase6_environment",
"phase7_colony_behavior", "phase8_full_integration")

foreach ($p in $phases) {
Write-Host "--- $p ---" -ForegroundColor Yellow

}
