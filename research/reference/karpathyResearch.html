Methodology

This report applies the Karpathy incremental research approach to complex scripts in the Smoke & Honey codebase. The core idea: never try to test or validate an entire complex system at once. Instead, isolate the smallest meaningful segment, verify it thoroughly, then expand scope by one increment -- always carrying forward what you already confirmed.

  • Start with the smallest unit that produces a verifiable output
  • Lock that down with assertions before adding the next layer
  • Each phase includes everything from prior phases plus one new concern
  • If a phase fails, the bug is isolated to the newly added segment
  • Continue until the full system is covered end-to-end

4. SaveManager.gd

610 lines 5 phases MED priority

Why This Script Needs Phased Testing

The save/load system is where game state meets serialization. A subtle bug here (e.g., base64 encoding dropping the last byte of a PackedByteArray, or a dictionary key mismatch) won't crash the game -- it will silently corrupt save data that only manifests as "my bees disappeared" or "my honey reset to zero" hours later. The two-phase loading architecture (load_from_disk -> apply_to_scene) means bugs could live in either phase. Testing the whole thing at once makes corruption nearly impossible to trace.

1 Primitive Serialization Round-Trip ~80 lines

Test that basic data types survive save/load: ints, floats, strings, bools, arrays, dictionaries. No game objects -- just JSON fidelity.

Test Targets

  • Integer values round-trip exactly (no float conversion artifacts)
  • Float values round-trip within acceptable precision (test edge cases: 0.0, -1.0, very large, very small)
  • String values with special characters survive (newlines, unicode if any, empty strings)
  • Nested dictionaries maintain structure and key names
  • Arrays maintain order and element types
  • SAVE_VERSION field is present and correct in output JSON
2 PackedByteArray Base64 Encoding ~60 lines

Test the critical base64 encoding/decoding path for PackedByteArrays. Cell data and cell age arrays go through this path -- a single byte error corrupts an entire frame.

Test Targets

  • PackedByteArray of known content (e.g., [0, 1, 2, ... 255]) round-trips to identical bytes
  • Empty PackedByteArray (size 0) round-trips without error
  • Large arrays (frame-sized: grid_cols * grid_rows * 2) round-trip correctly
  • Base64 string length is correct (4/3 ratio with padding)
  • No off-by-one: array[last_index] survives the round-trip
Cumulative scope: Phase 1 (primitives) + base64 byte arrays
3 Hive State Collection ~150 lines

Test the collect side: HiveSimulation state -> dictionary. This is the "save" direction. Every simulation field must be captured.

Test Targets

  • All HiveBox and HiveFrame data is captured (no missing frames or sides)

  • Cell arrays are base64 encoded (not raw arrays -- would bloat save file 10x)
  • Dictionary keys match the schema expected by apply_to_scene()

Cumulative scope: Phases 1-2 + hive state collection

Test Targets

  • After apply, HiveSimulation.population == saved population (exact match)
